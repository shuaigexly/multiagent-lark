# 飞书任务生成规范

## 核心原则：分析结论必须可落地

每个分析结论如果没有对应的具体行动任务，等于没有价值。
任务必须是 SMART（具体/可衡量/可执行/相关/有时限）。

## 任务提取规则

### 从分析中识别可行动点

优先提取以下类型结论为任务：
- **数字异常** → 排查任务（"DAU 下降 18% → 任务：本周排查 W3 推送策略变更"）
- **风险信号** → 预防任务（"Burn Rate 超标 → 任务：本月重审非核心支出"）
- **机会识别** → 利用任务（"竞品新融资 → 任务：下周完成差异化定位更新"）
- **决策需要** → 讨论任务（"方案A vs B → 任务：周五前 CEO 决策"）

### 不要生成的任务类型
❌ "持续关注 XX 指标"（没有截止时间）
❌ "了解竞品动态"（没有具体行动）
❌ "优化产品体验"（没有可衡量目标）

## 任务标准格式

每个任务必须包含：

```
任务标题：[动词 + 对象 + 量化目标]（≤20字）
优先级：P0（今日）/ P1（本周）/ P2（本月）/ P3（季度）
截止日期：基于当前分析时间计算
负责人：[从飞书通讯录/上下文中选择，或标注"待确认"]
任务描述：[背景1句 + 具体要做什么 + 完成标准]
关联文档：[如有飞书文档上下文，引用具体文档链接]
```

## 优先级分配规则

| 优先级 | 场景 | 截止时间 |
|--------|------|----------|
| P0 | 极高风险已触发、营收断崖、系统故障 | 今日/24小时内 |
| P1 | 高风险预警、KR严重落后、重要决策 | 本周内（5工作日） |
| P2 | 中等优化项、月度目标对齐 | 本月内（20工作日） |
| P3 | 长期改进、竞争布局、能力建设 | 本季度内 |

**规则**：每次输出任务中，P0+P1 不超过 3 个，避免优先级膨胀。

## 任务分配原则

优先使用飞书上下文中已有的人员：
- `feishu_context.tasks` 中出现过的 `assigned_to` 字段 → 对应负责人
- 任务清单中同类型任务的负责人 → 同类新任务的候选负责人

没有上下文时的默认分配规则：
- 数据类任务 → "数据分析师"
- 财务类任务 → "财务负责人"
- 产品类任务 → "产品经理"
- 执行类任务 → "运营负责人"
- 决策类任务 → "CEO / 负责人"

## 飞书任务 API 输出格式

当需要创建飞书任务时，按以下结构输出（便于程序解析）：

```json
{
  "tasks": [
    {
      "summary": "本周排查 W3 推送策略变更对留存的影响",
      "description": "W3 上线的推送策略调整后，Day7 留存从 32% 跌至 24%。需核查推送频率/内容/时机变更，确认是否回滚。",
      "due": "2025-01-17",
      "priority": "P1",
      "assignee_note": "数据分析师",
      "related_doc": "飞书文档链接（如有）"
    }
  ]
}
```

## 任务数量控制

| 分析规模 | 任务上限 |
|---------|---------|
| 单模块输出 | 3-5 个 |
| 多模块汇总 | 5-8 个 |
| 季度复盘 | 8-12 个 |

超出上限时，优先保留 P0/P1，其余合并或降级到文档跟进。

## 表述示例

✅ 好：
- "完成 8 月广告渠道 ROI 对比报告（截止 8/15，负责人：市场总监）"
- "下周一前决策是否暂停 B 渠道投放（预算影响：5 万/月）"

❌ 差：
- "分析广告效果"
- "考虑暂停某些渠道"
